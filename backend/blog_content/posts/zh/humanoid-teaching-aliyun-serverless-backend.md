# 仿真人教学 Aliyun Serverless：把正式后端模板先立住

仿真人教学主项目已经把 Render 演示版、课堂流程、教师端和学生端整理成了一个可运行的教育平台模板。`aliyun-serverless` 子项目则更像下一阶段的正式后端底座：它面向阿里云函数计算和 MySQL，把认证、资源、AI 教学能力、仿真课堂、统计和家长视图整理成一组 API。

这篇记录只基于低风险材料：该子项目的 `README.md` 和 `package.json`。它不读取源码全文，不展开数据库脚本，不发布环境变量值、密钥、部署配置、安装包、本地二进制文件或任何私有数据。

## 从演示后端走向正式后端

README 对这个子项目的定位很直接：这是仿真教学平台的正式后端模板，面向阿里云函数计算和 MySQL 部署。也就是说，它不是另一个页面 demo，而是把前端产品需要依赖的服务接口提前固定下来。

接口范围覆盖得比较完整：注册、登录、当前用户信息、教学资源列表、资源搜索、资源详情、AI 教案、AI 问答、错题复盘、学情分析、课堂知识库、课堂黑板、课堂对话、平台统计概览，以及家长端学生学情接口。对一个教学平台来说，这些接口刚好对应几条主线：用户身份、资源获取、AI 辅导、课堂交互、运营统计和家校协同。

这个拆法的价值在于，正式化不是等页面全部完成之后再补后端。相反，后端模板先把产品边界画出来，后续 uni-app、小程序、H5 或管理端都可以围绕这些 API 做接入，而不是各自临时拼接口。

## 课堂能力不是简单聊天

仿真课堂模块在 README 中被拆成了三层：`classroom_knowledge` 保存教研自有的板书摘要、关键点、例题和安全提示；黑板接口向前端返回课堂主页标题和当前知识点板书；课堂对话接口把学生问题、当前黑板内容、最近对话和 EMBER-Agent 安全约束交给大模型生成 AI 教师讲解。

这个设计没有把“AI 教师”简化成一个裸聊天接口。它先把课堂上下文、黑板内容、知识点和安全约束放进服务端，再由服务端调用模型。这样前端拿到的是受控课堂交互，而不是直接面对模型密钥和自由提示词。

README 还说明课堂对话会记录下来，便于后续课堂复盘和学情分析。这一点很关键。教学系统里的对话不是一次性消息流，它应该能回到学生画像、错题复盘、教师观察和家长反馈里。后端模板提前保留这条记录链路，说明它的目标是课堂业务闭环，而不只是生成一段回答。

## 密钥和外部资源放在服务端边界内

这个模板最值得肯定的边界，是它明确不把 DeepSeek Key 暴露给前端。AI 教案、在线问答、错题复盘和学情分析都应该经过服务端接口，这样才能统一处理密钥、日志、限流、安全提示和错误返回。

国家智慧教育平台资源也被放在服务端适配层里。README 列出了资源接口相关环境变量，包括 API base URL、API key、资源列表路径、详情路径、搜索路径和每分钟限流配置。正式路径、签名方式和限流规则以授权文档为准，拿到文档后再调整服务端请求头和路径映射。

这条原则比具体接口名更重要：国家平台资源原文不入库，数据库只保存用户、原创内容、访问日志、AI 调用日志和学情报告等自有数据。教育项目一旦接入官方资源，真正的工程风险往往不是“能不能请求成功”，而是资源是否被错误复制、密钥是否泄露、限流是否失控、调用行为是否可追踪。这个模板把这些问题放在服务端处理，是更稳的路线。

## 技术选型偏向小而清楚

`package.json` 显示这个后端模板是一个私有 Node ESM 项目，当前版本为 `0.1.0`。脚本很克制：`check` 用 `node --check` 做语法检查，`start:local` 用本地 server 脚本启动调试服务。

依赖也比较集中：`mysql2` 负责 MySQL 访问，`jsonwebtoken` 处理登录态，`bcryptjs` 处理密码哈希，`zod` 做输入校验，`dotenv` 管理本地环境变量，`uuid` 生成业务标识。它没有把模板阶段做成复杂框架堆叠，而是围绕 Serverless API、身份认证、数据库访问和参数校验保留必要组件。

本地启动流程也保持简单：安装依赖、准备环境变量文件、运行本地服务。README 说明本地接口默认运行在 `http://127.0.0.1:8787`，这对前端联调、接口验证和部署前自测都足够直接。

## 部署形态服务于多端产品

主项目文档已经提到未来会走 uni-app + 阿里云 Serverless + MySQL 的正式化路线。这个子项目就是这条路线里的后端模板。函数计算适合把 API 服务做成较轻的云端入口，MySQL 保存自有业务数据，前端可以继续演进成小程序、H5 或 App。

更重要的是，这个后端模板把“多端”需要的能力提前整理成 API，而不是把业务锁死在某个服务端页面里。资源列表、资源详情、AI 讲解、课堂黑板、课堂对话、统计概览和家长视图都可以被不同客户端复用。这样后续迁移时，页面可以换，客户端可以换，但后端的业务边界不必从零开始。

## 小结

`aliyun-serverless` 子项目的意义，是把仿真人教学从 Render 演示版继续推向正式后端模板。它覆盖了教育平台最核心的服务面：身份、资源、AI、课堂、统计和家长视图；同时把模型密钥、国家平台资源适配、限流和数据保存边界都放在服务端。

它还不应该被描述成完整上线后的商业后端。更准确的说法是：这是一个面向正式部署路线的后端骨架，先把 API、数据边界和外部服务适配位置固定下来。对一个需要同时服务学生端、教师端、家长端和 AI 教学能力的项目来说，这一步比继续堆页面功能更关键。
