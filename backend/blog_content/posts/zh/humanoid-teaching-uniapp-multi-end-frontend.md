# 仿真人教学 uni-app：把课堂产品做成多端前端模板

`uniapp` 子项目是仿真人教学平台的多端前端模板。主项目已经验证了教学平台的基础流程，`aliyun-serverless` 子项目把正式后端 API 边界立了起来，而这个前端模板负责承接学生、教师、课堂、家长和 AI 教学能力，把它们组织成可继续扩展到 H5、小程序和 App 的客户端形态。

这篇记录只基于低风险材料：该子项目的 `README.md` 和 `package.json`。它没有读取页面源码、环境变量样例、构建产物、账号信息、私钥、数据库、安装包或本地二进制文件。

## 多端不是简单换壳

README 给出的定位是“仿真教学平台的多端前端模板”，支持 H5、小程序和 App 方向继续扩展。这里的重点不只是使用了 uni-app，而是把教学产品的页面边界提前整理成一组稳定入口。

页面列表覆盖了完整学习闭环：登录和快速注册、国家平台资源筛选展示、仿真课堂、智能备课、在线答疑、错题复盘、家校协同。也就是说，这个模板不是只做一个课堂页面 demo，而是在客户端层面先画出学生端、教师端、AI 辅助和家长端之间的产品地图。

这种拆法对多端迁移很重要。H5、小程序和 App 的运行环境不同，但用户任务不会因为端不同而消失。学生仍然需要进入课堂、看资源、提问和复盘；教师仍然需要备课；家长仍然需要看到学习状态。前端模板先把这些入口稳定下来，后续再适配不同平台能力时，才不会把业务流程打散。

## 仿真课堂是前端核心场景

课堂页是这个模板里最明确的核心场景。README 写明，黑板主页固定展示“基于EMBER-Agent安全增强的仿真课堂”，并支持知识库卡片切换。选中知识库后，黑板板书会自动刷新，AI 教师讲解区也会围绕课堂上下文工作。

这说明课堂不是一个普通聊天窗口。它至少包含四个前端状态：当前知识库、黑板内容、AI 教师讲解、学生与 AI 教师的对话。把这些状态放在同一个课堂页面里，前端就能表达“正在上课”的感觉，而不是把教学拆成几个互不相干的工具按钮。

README 还提到 H5 端系统语音播报，以及语音提问入口预留。这个边界处理得比较务实：当前模板先给 H5 留出可运行的系统语音播报路径，同时把阿里云或讯飞语音转文字作为后续接入点。对多端项目来说，语音能力往往和平台权限、SDK、审核要求有关，先保留入口而不强行绑定某个实现，是更稳的工程路线。

## 页面组合对应教育业务链路

从页面清单看，这个前端模板把教育业务拆成几条并行链路。

第一条是资源链路：`pages/home/index` 负责国家平台资源筛选展示。它应该对接后端的资源列表、搜索和详情能力，让学生或教师能从官方资源入口进入学习材料。

第二条是课堂链路：`pages/classroom/index` 承担仿真课堂、黑板、知识库、AI 教师讲解、对话和语音入口。这是项目最有辨识度的交互场景，也是 EMBER-Agent 安全增强最需要被前端清楚表达的位置。

第三条是 AI 学习辅助链路：`pages/lesson-prep/index` 对应 DeepSeek 智能备课，`pages/qa/index` 对应在线答疑，`pages/wrongbook/index` 对应错题复盘。它们不是孤立功能，而是把课前准备、课中提问和课后复盘连接起来。

第四条是家校协同链路：`pages/parent/index` 给家长端留出入口。对教学产品来说，这个页面的存在很重要，因为它说明学习结果不仅服务学生和教师，也会回到家庭侧的观察与反馈。

## 技术栈保持轻量

`package.json` 显示这个子项目是一个私有的 Node ESM 项目，包名为 `humanoid-teaching-uniapp`，版本为 `0.1.0`。脚本集中在四个方向：H5 开发、H5 构建、微信小程序开发、微信小程序构建。

依赖也很集中：`@dcloudio/uni-app`、`@dcloudio/uni-components`、`@dcloudio/uni-h5` 和 Vue 3 组成运行基础；开发依赖使用 uni 相关类型、CLI shared 包、Vite uni 插件和 Vite 5。这个组合符合模板阶段的目标：先让多端前端能跑起来，再逐步把具体业务页、平台能力和后端接口接进去。

本地运行流程也保持简单：进入 `uniapp` 目录、安装依赖、准备环境变量文件、运行 H5 开发命令。README 说明 API base URL 在本地指向开发服务入口，生产环境则应切换到阿里云 API 网关或函数计算 HTTP 触发器域名。这个说明把前后端边界讲清楚了：前端不直接拥有正式服务地址和密钥，它通过环境配置连接后端入口。

## 和 Serverless 后端形成配套

前一篇后端模板文章里提到，`aliyun-serverless` 子项目把身份、资源、AI 教案、AI 问答、错题复盘、学情分析、课堂黑板、课堂对话、统计和家长端接口整理成服务边界。`uniapp` 子项目正好是这些接口的多端承接层。

这两个子项目放在一起看，路线很清楚：后端负责密钥、官方资源适配、AI 调用、数据保存和安全约束；前端负责页面组织、课堂交互、语音入口和多端展示。这样的分工避免了把 AI 密钥、资源授权细节或课堂安全策略塞进客户端，也避免了让后端关心每个端的页面交互细节。

对教育软件来说，这个边界比页面数量更关键。学生端和家长端可能先从 H5 启动，后续再进入小程序；教师端也可能先做轻量版本。只要 API 和页面入口稳定，产品就能在不同客户端之间逐步迁移。

## 小结

`uniapp` 子项目的意义，是把仿真人教学的产品形态从单一 Web 演示继续推向多端客户端模板。它把登录、资源、仿真课堂、智能备课、在线答疑、错题复盘和家校协同放进统一前端结构里，并为 H5、小程序和 App 扩展保留了技术路径。

它目前更适合被描述为“多端前端骨架”，而不是完整上线的客户端。它已经把页面边界、课堂核心交互、语音扩展位置和后端 API 接入方式摆清楚了。接下来真正重要的工作，是让这些页面继续对齐正式 Serverless 后端，把课堂数据、AI 回复、资源筛选和学习反馈变成可验证的端到端流程。
