# 仿真人教学：从 Render 演示版走向多端教学平台模板

仿真人教学这个项目值得记录的地方，是它没有只停留在“教学页面 demo”，而是把学生端、教师端、仿真课堂、题库、部署和后续迁移路线都整理成了一个可继续扩展的教学软件模板。README 把当前版本定位得很明确：这是一个可以直接部署到 Render 的 Node + Express + EJS + PostgreSQL 基础模板，同时也为 uni-app、阿里云 Serverless、MySQL、DeepSeek API 和国家智慧教育平台资源适配预留了迁移方向。

这篇记录只基于低风险材料：`README.md`、`package.json`，以及 `docs/` 下的公开说明文档。它不展开源码实现，不发布本地工程路径、私有仓库细节、环境变量值、数据库内容、安装包或二进制文件。

## 先把教学闭环跑起来

当前模板的第一层价值，是把教学系统的主流程先做成可以运行的闭环。README 中列出的能力覆盖学生注册登录、家长授权、会员准入、教师注册、教师后台、学生名单、派题视图、个性化首页、学情画像、真实题库接入、练习包派发、课程路径、VIP 权益、教研硬件生态页，以及学生端和教师端 JSON API。

这说明项目不是单页展示，而是按“教学平台”的产品边界在组织。学生侧需要建档、诊断、练习和课堂记录；教师侧需要看学生、派题、查看课堂；平台侧还要考虑会员、课程、资源、家校协同和部署运维。即使当前仍是模板阶段，这些边界已经把后续扩展的方向固定下来。

技术栈也偏向可部署的常规组合：Node.js 20+、Express、EJS、PostgreSQL，并且在没有数据库时可以切换到本地内存模式，便于快速演示。这种选择的重点不是炫技，而是先让课堂业务可以被启动、验证和交付。

## 仿真课堂有明确契约

`docs/simclass-api-contract.md` 把仿真课堂拆成了一组可复用接口，而不是把课堂行为绑死在某个页面里。学生端可以获取课堂列表、查看单节课堂、开启课堂、提交答案或追问、结束课堂；教师端可以查看课堂看板，并按状态或学生过滤记录。

接口统一返回 `{ ok, data, meta }` 或 `{ ok, error }` 的结构，这对后续接入小程序、移动端或 uni-app 很关键。前端不需要猜后端行为，只要按契约处理成功、失败、权限和课堂状态即可。

课堂流程本身也有清楚规则：开课时读取学生学情画像，自动生成报到记录，从真实题库或保底题库选择首题；答对时推进下一题，答错时保留当前题并生成纠错反馈；只追问不计分，空提交只提醒继续作答。这个细节很重要，因为仿真课堂不能只是一个“AI 老师聊天框”，它还要保留教学系统里的题目、作答、反馈、统计和复盘逻辑。

## 黑板主视觉被测试锁住

交付报告和 API 契约都反复强调一个产品要求：仿真课堂黑板主视觉必须固定显示“基于EMBER-Agent安全增强的仿真课堂”，并且黑板内只保留课堂问题、答案选项和提交按钮。知识库来源、今日板书、AI 教师卡片等辅助信息不能挤进黑板。

这个要求已经被自动测试固化。`package.json` 中的 `test:classroom` 用来验证黑板标题、题目、选项、提交按钮和空状态；`test:classroom-flow` 验证开课、答对、答错、追问、空提交等课堂流程；`test:auth-boundaries` 检查学生端、教师端和公开接口的权限边界；`test:store-contract` 检查用户、登录态、学情、派题和课堂记录的存储主链。

最后，`test:simclass` 把这些验证合在一起，成为仿真课堂模块的基础回归门槛。这个做法比“页面看起来对”更可靠：后续任何人改课堂页、接口或存储层，都可以用同一组命令检查是否破坏了产品契约。

## 上线不只看能不能启动

项目文档没有把部署写成一句“可以上 Render”，而是把生产检查拆成了可执行的自检。`docs/simclass-production-readiness.md` 说明 `verify:production` 会检查 `package.json`、Render Blueprint、环境变量说明、健康检查、Postgres 持久化、生产 Cookie 设置和核心验证脚本是否保留。

这类检查解决的是模板项目常见问题：本地能跑，但生产配置在迭代中被误删；健康检查还在，但无法说明存储模式和题库来源；环境变量存在，却没有明确哪些必须由平台生成。把这些规则做成自检命令，可以在上线前提前发现配置漂移。

文档里还区分了必需检查和可选检查。本地 runner 每轮会执行生产自检和仿真课堂总回归，也会尝试检查本地健康地址和 Render 新鲜度；可选检查失败时需要关注，但不等同于本地业务回归失败。这个区分让运维状态更容易解释。

## 迁移路线已经预留

当前 Render 版本是演示和过渡环境，项目同时准备了正式架构模板：uni-app 负责小程序、H5 和 App；阿里云函数计算和 API 网关负责后端；MySQL 保存业务数据；DeepSeek API 负责智能备课、在线答疑、错题复盘和学情分析；国家智慧教育平台资源只做授权调用和展示。

`docs/uniapp-aliyun-serverless-blueprint.md` 里最关键的不是技术名词，而是资源合规原则：国家平台资源不把原文、视频、完整解析落到自有库，只保存资源 ID、来源、访问日志、收藏和学习行为等业务元数据；资源展示要标注来源，资源详情按需实时调用，并通过服务端限流控制访问。

这条边界很重要。教育平台一旦接入外部题库、官方资源和大模型能力，真正的风险往往不在页面，而在数据保存、授权调用、密钥暴露和资源二次分发。文档提前把这些限制写入迁移方案，说明项目在模板阶段就开始区分“可以展示”和“可以存储”的边界。

## 小结

仿真人教学目前更像一个面向智能教学场景的工程模板：Render 版本负责快速演示和上线闭环，仿真课堂契约负责稳定核心体验，自动测试负责防回归，生产自检负责部署前检查，而 uni-app + 阿里云 Serverless 蓝图负责后续多端正式化。

它还不是一个完整商业教学平台，也不应该被包装成已经完成全部功能的产品。更准确的说法是：这个项目正在把“仿真课堂 + 教师端 + 学生端 + 题库 + AI 能力 + 部署运维”先整理成可运行、可验证、可迁移的模板。这个阶段的价值，在于每个后续模块都有比较清楚的落点，而不是继续堆在一个无法验收的教学 demo 里。
