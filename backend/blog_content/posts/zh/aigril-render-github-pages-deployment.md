# AIGril 的上线方式：GitHub Pages 前端加 Render 后端

AIGril 不是只停在本地演示里的虚拟人项目。它已经被拆成了一个可以在线体验的前后端系统：前端放在 GitHub Pages，后端放在 Render，用户打开网页就能看到 AIGL 角色并尝试对话。

这篇文章记录的是当前项目最核心的一条工程主线：怎样把一个包含 3D 角色、流式对话、记忆服务和安全接口的 AI 应用，整理成可以被别人访问的在线作品。

## 为什么选择前后端分开部署

AIGril 的前端主要负责体验。

它包括：

- 3D VRM 角色渲染
- 动作和表情系统
- 流式文本展示
- 口型和说话状态
- 浏览器端交互界面

这些内容很适合部署在 GitHub Pages 上。它是静态站点，不需要服务器常驻，也很适合展示项目。

后端则负责不适合暴露在浏览器里的能力：

- 大模型接口调用
- 会话记忆
- RAG 查询
- 安全检测 API
- TTS 或其他需要密钥的服务
- SQLite 和后端状态管理

这部分需要一个真实的 Python 服务，所以我把它部署在 Render 上，用 FastAPI 对外提供接口。

## 当前在线体验入口

现在项目主要有三个入口：

- 完整体验地址：[https://haowenGuo.github.io/AIGril/?backend=https://airi-backend.onrender.com](https://haowenGuo.github.io/AIGril/?backend=https://airi-backend.onrender.com)
- 前端静态页面：[https://haowenGuo.github.io/AIGril/](https://haowenGuo.github.io/AIGril/)
- 后端 API 文档：[https://airi-backend.onrender.com/docs](https://airi-backend.onrender.com/docs)

这个设计的好处是，访问者不需要安装任何东西，就能先快速理解项目是什么；如果想继续研究源码，再去看 GitHub 仓库和后端接口。

## Render 后端承担什么

AIGril 的后端不是一个简单的转发层。

它至少承担了几类职责：

- 提供 `/api/chat` 一类对话接口
- 管理用户会话和消息历史
- 在对话变长后做记忆压缩
- 提供内容安全检测能力
- 为后续语音、教学平台、博客等扩展模块提供统一 FastAPI 服务

我选择 Render 的原因很直接：它对 FastAPI 这种项目足够友好，绑定 GitHub 后可以自动根据 `main` 分支重新部署。对一个需要展示给别人看的个人项目来说，这比自己维护云服务器轻很多。

当然，它也有代价。免费或低配实例会有冷启动，跨区域访问大模型 API 时延也会影响体验。所以后面我把高延迟的 TTS 暂时放下，优先回到流式文本输出，让交互先稳定。

## GitHub Pages 前端承担什么

前端这边的目标是把“项目能力”变成“用户能感知到的体验”。

AIGril 的页面不是普通聊天框，而是围绕 AIGL 这个 3D 角色组织的：

- 用户输入消息
- 后端返回流式文本
- 角色进入说话状态
- 文本逐步显示
- 口型按文本节奏做 fallback lip sync
- 表情和动作根据接口状态切换

这部分放在 GitHub Pages 上很合适，因为它可以作为项目主页，也可以作为作品集入口。别人点开链接，就能直接看到角色、页面、交互和整体完成度。

## 源码和安装包的处理

源码目前放在 GitHub 仓库中：

[https://github.com/haowenGuo/AIGril](https://github.com/haowenGuo/AIGril)

桌面端也已经具备 Electron 打包链路。根据项目 README，Windows 桌面包可以通过下面命令生成：

```bash
pnpm desktop:package
```

生成结果会进入 `release/` 目录，包括安装版、便携版和解压运行版。

不过我不会在自动博客任务里直接上传本机安装包，因为这一步需要确认版本、体积、依赖和是否适合公开分发。更稳妥的方式是先在文章里说明打包方式，等版本稳定后再放 GitHub Release 下载链接。

## 这套部署方式带来的变化

完成部署后，AIGril 的性质发生了变化。

它不再只是“我本机能跑的项目”，而是变成了：

- 可以通过网页访问的作品
- 可以给同学或面试官演示的项目
- 可以持续写博客记录迭代的产品
- 可以继续扩展后端 API 的平台

这也是为什么我后面又补了博客系统和发文规范。项目一旦能在线访问，就不只是代码仓库了，它需要文档、展示、迭代记录和对外表达。

## 下一步

接下来更值得做的不是盲目堆功能，而是继续把体验闭环打磨清楚：

- 降低后端冷启动和模型调用延迟
- 优化流式输出和角色说话状态
- 把桌面端安装包整理成可公开下载的 Release
- 继续完善博客，把每次迭代沉淀成文章
- 明确哪些源码、素材和模型资源可以公开分发

AIGril 当前的部署方案并不复杂，但它已经把一个本地 AI 虚拟人项目变成了一个能被别人打开、体验、理解和继续关注的在线作品。
